Slide 2 Project Overview & Motivation ~1 min

LA has over 30,000 restaurants, but Google ratings are almost useless for filtering — 89% of places are rated 4 stars or above. The real signal is the 4.5-star threshold, and only 41% of restaurants actually reach it.

So the question I wanted to answer is: can we predict whether a restaurant crosses that threshold using only publicly available metadata — neighborhood, cuisine type, price level, and review count?

My approach was to collect data from the Google Places API across 20 LA neighborhoods, train a binary classifier on that data, and serve predictions through a FastAPI backend and a Streamlit frontend, both deployed on Google Cloud Run.

Slide 3 Data Collection & Preprocessing ~1.5 min

I used the Google Places API's searchNearby endpoint to query 20 LA neighborhoods. The API caps results at 20 per call, so to get better coverage I used 5 search points per neighborhood — the center plus offsets north, south, east, and west. After de-duplicating by place ID, I ended up with 2,443 unique restaurants.

For preprocessing: 26% of restaurants were missing a price level, so I imputed with the median. Review count was heavily right-skewed — ranging from 1 to 28,000 — so I log-transformed it. And I created the binary target: high-rated equals 1 if the rating is 4.5 stars or above.

A few EDA highlights you can see in the charts: Little Tokyo has the highest rate of high-rated restaurants at 63%, Monterey Park the lowest at 24%. And higher price tiers strongly correlate with higher ratings — very expensive restaurants are high-rated 75% of the time versus 36% for budget spots.

Slide 4 Model Development & Evaluation ~2 min

I compared five models: Logistic Regression, Gradient Boosting, Random Forest, and two MLP neural networks. The metric I cared most about was recall on the high-rated class — missing a great restaurant is a worse outcome than a false alarm.

Gradient Boosting had 65% accuracy but a recall of only 0.29 — it was basically predicting "not high-rated" for everything. The MLP models were even worse, with the larger network hitting a recall of 0.18. Neural networks underperform here for two reasons: the dataset is only 2,400 rows, and the features are mostly categorical — tree methods handle one-hot encoded data natively, and the MLP had no mechanism for class imbalance.

Random Forest with class_weight="balanced" was the winner — recall of 0.64 and the best AUC at 0.652. The balanced weighting upweights the minority class during training, which is what rescued the recall. And to be clear, its 57% accuracy isn't bad — a model predicting "not high-rated" for every single restaurant would score 59%.

On feature importance: place type accounts for 63% of the model's decisions. Types like convenience stores are never high-rated; Italian restaurants are 54% of the time. Review count is the strongest individual feature at 15%. Neighborhood and price matter but less so.

Slide 5 Solution Architecture ~1 min

The top section of the diagram is the offline pipeline — run once locally. I scraped the data, saved it as a CSV, and trained a scikit-learn Pipeline that bundles the preprocessor and the classifier into a single model.pkl file.

That artifact gets baked directly into the FastAPI container — no separate model store needed. I built both images with Podman targeting Linux AMD64, pushed them to Google Artifact Registry, and deployed two independent Cloud Run services. The Streamlit app calls the API's /meta endpoint on load to populate the dropdowns, and /predict on every button click. The API URL is injected as an environment variable at deploy time, keeping the two services fully decoupled.

Slide 7 Challenges, Learnings & Future Work ~1 min

Three main technical challenges. Class imbalance — solved with class_weight="balanced", which doubled recall from 0.29 to 0.64. Cross-platform containers — my laptop is Apple Silicon but Cloud Run needs Linux AMD64, handled with Podman's --platform flag. And the API's 20-result cap per call — worked around with 5 search points per neighborhood.

On AI assistants — I used Claude Code throughout. It was strongest at scaffolding: it generated the full FastAPI setup, Streamlit app, Dockerfiles, and test suite from a single description, and handled all Cloud Run deployment. Where human judgment was still essential was deciding which features to use, interpreting what the data means about LA dining, and evaluating whether the model's behavior made real-world sense. The lesson: always review AI-generated code — it's fast but it doesn't understand your domain.

For future work: I'd most want to add SHAP explanations in the UI so users can see why a prediction was made, and a map view showing prediction probabilities across LA neighborhoods. Thank you — happy to take questions.
